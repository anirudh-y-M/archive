## Key Concepts

- **Three observability pillars**: Metrics (Prometheus), distributed tracing (Jaeger/Zipkin/OTel), and access logs -- all generated automatically by Envoy sidecars with zero application code changes.

- **Standard metrics**: `istio_requests_total`, `istio_request_duration_milliseconds`, `istio_request_bytes`, `istio_response_bytes`, plus TCP counters. Generated by the `istio.stats` Wasm filter. Exposed on port 15090.

- **Distributed tracing**: Envoy generates spans per request. Spans are linked by trace headers (B3 or W3C). **Critical caveat**: applications MUST propagate trace headers between inbound and outbound calls, or traces appear disconnected.

- **Kiali**: Observability console. Service topology graph, traffic animation, Istio config validation, health indicators. Pulls from Prometheus + K8s API + optionally Jaeger.

- **WasmPlugin**: Load WebAssembly modules into Envoy as HTTP filters. Compiled from Go/Rust, stored in OCI registry, deployed via CRD. Sandboxed, portable, survives Istio upgrades. ~10-50 microsecond overhead.

- **EnvoyFilter**: Low-level escape hatch that directly patches generated Envoy config. Brittle across Istio versions -- use only as last resort.

- **Telemetry API**: CRD for configuring metrics, tracing, and access logging per-workload/namespace/mesh. Scoping: workload > namespace > mesh.

## Quick Reference

```
Data Flow:
  Envoy ──:15090──> Prometheus ──> Grafana (dashboards)
  Envoy ──spans──> Jaeger/OTel Collector ──> Jaeger UI
  Envoy ──logs──> stdout / gRPC ALS ──> Loki/Elasticsearch
  Prometheus + K8s API ──> Kiali (topology graph)
```

| Extension Method | Use When | Risk Level |
|-----------------|----------|-----------|
| Telemetry API | Custom metrics, tracing config, log filtering | Low (stable API) |
| WasmPlugin | Custom logic (headers, validation, metrics) | Low-Medium |
| Lua (via EnvoyFilter) | Quick one-off customizations | Medium |
| EnvoyFilter | Envoy features not exposed by any CRD | High (version-fragile) |

**Headers apps must propagate for tracing:**
`x-request-id`, `x-b3-traceid`, `x-b3-spanid`, `x-b3-sampled`, `traceparent`, `tracestate`

## Key Takeaways

- Metrics and access logs are automatic. Distributed tracing requires applications to propagate trace context headers -- without this, multi-hop traces break into disconnected single spans.
- Prefer WasmPlugin over EnvoyFilter for custom Envoy extensions -- it uses a stable ABI and survives Istio upgrades.
- EnvoyFilter patches reference internal Envoy config that changes between Istio versions. They can silently fail with no error.
- The Telemetry API lets you customize metrics dimensions, disable specific metrics, configure sampling rates, and filter access logs -- all declaratively per workload or namespace.
